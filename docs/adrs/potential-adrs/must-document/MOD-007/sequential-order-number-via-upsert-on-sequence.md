# Sequential orderNumber Generation via Upsert on orderNumberSequence

## Score
Total: 118/150 | Step 0: 75 | Scope+Impact: 10/25 | Cost to Change: 15/25 | Team Knowledge: 18/25

## Evidence
- File: `prisma/schema.prisma` — `OrderNumberSequence` model with singleton row (`id: Int @id @default(1)`, `nextValue: Int @default(1)`)
- File: `src/modules/orders/order.service.ts` — `reserveOrderNumber(tx)` calls `tx.orderNumberSequence.upsert({ where: { id: 1 }, update: { nextValue: { increment: 1 } } })`; formats result as `ORD-000001` with zero-padding

## Decision Summary
Human-readable, sequential order numbers (`ORD-000001`, `ORD-000002`, …) are generated by atomically incrementing a singleton counter row in the `order_number_sequences` table using Prisma's `upsert` with `increment`. The operation runs inside the order creation `prisma.$transaction`, ensuring the sequence value is committed if and only if the order is created successfully. The UUID primary key serves as the system identifier; the `orderNumber` is a display-only, user-facing identifier.

## Context
UUID primary keys are not user-friendly for order references in customer communications, support tickets, or invoices. A sequential, human-readable order number is a UX requirement. Using MySQL's `AUTO_INCREMENT` on the orders table itself was considered but rejected because it would not allow zero-padding or prefix formatting without a trigger.

## Alternatives Considered
- **MySQL AUTO_INCREMENT on orders table:** The formatted `ORD-XXXXXX` string cannot be derived from a plain AUTO_INCREMENT ID without additional logic; gaps caused by rollbacks are more visible to end users with this approach.
- **UUID-based order number:** Not human-readable; unsuitable for customer-facing display.

## Consequences
**Positive:**
- Human-readable, sequential order numbers with consistent zero-padded format
- Atomic increment inside the creation transaction: no order is created without a sequence number, no sequence number is consumed without an order
- `upsert` handles the first-run case (creates the singleton row if absent) without a separate migration seed

**Negative / Trade-offs:**
- The singleton row is a write contention point under high concurrent order creation; at scale, this becomes a bottleneck
- Gaps in the sequence are possible if the creation transaction rolls back after the upsert but before the final commit (e.g., stock validation failure)
