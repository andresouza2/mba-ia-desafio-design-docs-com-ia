## Decision Summary
Pino is the sole logging library for the entire project (API and planned worker). A single `logger` singleton is exported from `src/shared/logger/index.ts` and imported by all consumers. Logs are structured JSON in production (for log aggregators) and human-readable via `pino-pretty` in development. A security-critical redaction list prevents sensitive fields (`authorization`, `password`, `passwordHash`, `token`, `accessToken`) from appearing in any log output. HTTP request logs include a correlation `requestId` (UUID v4) generated by the request-logger middleware.

## Context
Structured JSON logs are required for machine-readable log aggregation (e.g., Datadog, CloudWatch). Unstructured `console.log` output is not queryable at scale. Redaction is a security requirement: auth tokens and password hashes must never appear in logs regardless of which code path logs the request object.

## Alternatives Considered
- **Winston:** Heavier, slower, less structured by default; requires plugins for JSON output and redaction.
- **console.log:** No structure, no log levels, no redaction, not aggregatable.
- **Morgan (HTTP-only):** Covers HTTP request logging only; no application-level structured logging or redaction.

## Consequences
**Positive:**
- Sub-millisecond logging overhead (Pino is the fastest Node.js logger)
- Security redaction is configured once and applies to all log calls automatically
- `requestId` enables log correlation across the full request lifecycle
- Worker process reuses the same logger with zero additional configuration

**Negative / Trade-offs:**
- Redaction paths use glob patterns (`*.password`) — overly broad patterns might redact legitimate fields; too narrow patterns might miss sensitive keys
- `pino-pretty` is a dev dependency; production logs are raw JSON and require tooling to read interactively
